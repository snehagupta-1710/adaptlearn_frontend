Types of Keys in a Database

In relational databases, keys are essential constructs that enforce uniqueness, referential integrity, and efficient data retrieval. Keys not only identify individual records but also define relationships between tables, preventing redundancy and ensuring consistency across datasets.

1. Primary Key

A Primary Key is a column or combination of columns that uniquely identifies a record in a table. Each primary key must contain unique, non-null values. Only one primary key is allowed per table, but it can be composite if needed.

Example: In a Students table:

  • Student_ID = 101
  • Name = "Riya"
  • Class = "5th"
The Student_ID uniquely identifies each student and serves as the primary key.

2. Foreign Key

A Foreign Key is a field (or set of fields) that establishes a link between two tables. It references a primary key in another table, ensuring that relationships are maintained and preventing orphan records.

Example: In an Enrollments table:

  • Enrollment_ID = 1
  • Student_ID = 101 (Foreign Key referencing Students table)
  • Course_ID = 202 (Foreign Key referencing Courses table)
This ensures each enrollment corresponds to a valid student and course.

3. Candidate Key

A Candidate Key is any column or set of columns that can uniquely identify a record. A table can have multiple candidate keys; one is chosen as the primary key, and the others become alternate keys.

Example: In a Students table:

  • Student_ID = 101
  • Email = "riya@gmail.com"
Both columns can uniquely identify a student, making them candidate keys.

4. Composite Key

A Composite Key is formed by combining two or more columns to create a unique identifier. This is useful when no single column can ensure uniqueness.

Example: In an Enrollments table:

  • Student_ID = 101
  • Course_ID = 202
Together, Student_ID + Course_ID forms a composite key, ensuring that a student cannot enroll in the same course multiple times.

5. Super Key

A Super Key is any combination of columns that can uniquely identify a record. All primary keys are super keys, but not all super keys are minimal.

Example: In the Students table, Student_ID + Email or Student_ID + Name can also serve as super keys. These combinations uniquely identify a student but may include unnecessary columns.

6. Alternate Key

An Alternate Key is a candidate key not chosen as the primary key. It can still uniquely identify records and provide an alternative access path for queries.

Example: If Student_ID is the primary key, Email can be an alternate key.

Why Keys Are Important

  • Prevent duplication and maintain data integrity.
  • Establish and enforce relationships between tables.
  • Facilitate efficient data retrieval and queries.
  • Support accurate updates and deletions across related tables.
  • Ensure reliable database design for scalability and performance.

Example: School Database Schema

  • Students Table: Student_ID (Primary Key), Name, Email (Candidate/Alternate Key)
  • Courses Table: Course_ID (Primary Key), Course_Name
  • Enrollments Table: Enrollment_ID (Primary Key), Student_ID (Foreign Key), Course_ID (Foreign Key), Student_ID + Course_ID (Composite Key)

This schema shows how primary, foreign, candidate, composite, and alternate keys work together to enforce integrity and maintain efficient relationships across tables.

Practical Tip

Plan your keys carefully before creating tables. Use composite keys for many-to-many relationships and alternate keys to optimize queries. Proper key design ensures data accuracy, efficient retrieval, and stable database structure.
